## 2024-06-25 - Expensive String Parsing in Hot Loops
**Learning:** Found `datetime.strptime()` being used inside a 30 FPS high-frequency computer vision hot loop (`detection_loop` in `fastapi_server.py`) to parse previously formatted string timestamps just to calculate time deltas for rate-limiting. This is an extremely expensive operation (string parsing vs basic float math) that adds unnecessary overhead per frame. Additionally, the new alert string was being formatted before the rate-limit check was performed, meaning `datetime.now().strftime()` was executing on every frame while an alert condition was active, even if the alert was going to be discarded by the cooldown.
**Action:** When implementing rate limiting or throttling inside hot loops (especially frame processing loops), always use native float timestamps (`time.time()`) and pre-calculated deltas for time tracking. Defer all computationally expensive string formatting (`strftime`) until *after* the conditional checks/rate limits have passed.
