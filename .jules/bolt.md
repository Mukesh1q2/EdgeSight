## 2025-04-28 - Optimize alert throttling to remove expensive string parsing in hot loop
**Learning:** Found computationally expensive string operations (`datetime.strptime()` and `datetime.now().strftime()`) running on every frame inside the ~30 FPS `detection_loop` when the fall probability exceeds the threshold. This caused unnecessary overhead because these strings were only needed if an actual alert was added, not checked on every single frame during an event. String formatting and parsing are notoriously slow in Python, especially in a computer vision hot loop.
**Action:** Use fast, native float comparison with `time.time()` (e.g. `current_time - last_alert_time`) for throttling/rate limiting, and defer all expensive string formatting operations inside conditional blocks so they are only called when they actually trigger.
